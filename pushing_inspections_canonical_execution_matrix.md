# PushingInspections Canonical Execution Matrix

## Purpose

`PushingInspections` is the evidence lane for the automotive control plane.

Its job is to convert uncertainty into condition truth:

- what the asset is
- what is wrong
- what is needed
- what can wait
- what the next downstream lane should be

This lane feeds:

- parts
- repair
- transport
- title / DMV readiness
- finance or insurance decisions when condition affects value or risk

## Operating Truth

`PushingInspections` is not a standalone namespace-backed object yet.

There is no `pushingInspections.json` shell in the live export, so the lane currently exists as a control concept spread across real PCRM objects and operator surfaces.

The right operating model is:

`public or operator surface -> intake shell -> inspection evidence shell -> execution ticket -> micro tasks -> downstream lane -> memory writeback`

The field leg can happen through subcontractors, shops, inspectors, or remote evidence capture.
The control leg stays in PCRM.

## Object Stack

### 1. Intake shell

Primary object:
- `p242835887_service_requests`

Job:
- capture the raw concern before the lane decides what kind of inspection it is

Strong live example:
- `177800129272` = `CORVETTE NOT STARTING`

Required facts:
- asset identity
- concern statement
- urgency
- location
- contact
- inspection intent

### 2. Inspection evidence shell

Primary object:
- `p242835887_vehicle_inspections`

Job:
- hold the inspection record itself
- capture the evidence state and findings state

Current reality:
- live and populated
- many rows are still skeletal and auto-generated by automation

### 3. Itemized findings shell

Primary object:
- `p242835887_inspection_items`

Job:
- break the inspection into actionable findings
- let downstream lanes consume specific issues rather than one giant blob

Current reality:
- exists, but is still barely populated

### 4. Adjacent evidence shell

Primary object:
- `p242835887_credit_inspections`

Job:
- represent the adjacent finance/credit inspection pattern

Important note:
- this is a real inspection family, but it should stay conceptually separate from the automotive inspection lane

### 5. Execution shell

Primary object:
- `tickets`

Job:
- own operational control when the inspection produces work

Current inspection-facing ticket pipeline:
- `Auto Repair`
- pipeline id `1428504282`

Strong live examples:
- `253969998545` = `Auto Repair - Nexon EV - Body Inspection`
- `253351797461` = `Auto Repair - Ahmed Ismaeil - Vehicle Repair`

### 6. Micro-follow-through shell

Primary object:
- `tasks`

Job:
- absorb missing facts, callback chase, photo gaps, retest needs, shop chase, and closeout follow-through

## Canonical Workflow

Workflow name:
- `pushing_inspections_evidence_orchestration_v1`

Workflow intent:
- take an asset from raw concern to verified findings, then route cleanly into repair, parts, transport, title/DMV readiness, or closeout

### Workflow phases

#### 1. `ingest`

Purpose:
- capture the raw concern and normalize it into inspection intent

Primary record:
- `p242835887_service_requests`

Required outputs:
- service request exists
- asset identity is usable
- inspection type and urgency are legible

Primary owner:
- `google_run_brain_ingestion`

Supporting owner:
- `pushingcap_orchestrator`

#### 2. `bind`

Purpose:
- create the inspection shell and connect it to the intake shell

Primary record:
- `p242835887_vehicle_inspections`

Required outputs:
- inspection object exists
- inspection id is stable
- inspection type is classified

Primary owner:
- `pushingcap_orchestrator`

#### 3. `qualify`

Purpose:
- gather first-through-the-door facts and decide whether the lane can proceed

Primary records:
- `p242835887_service_requests`
- `p242835887_vehicle_inspections`

Required outputs:
- VIN / address / photos / signature / consent are usable enough to proceed

Primary owner:
- `pushingcap_orchestrator`

#### 4. `inspect`

Purpose:
- gather evidence and turn it into itemized findings

Primary records:
- `p242835887_vehicle_inspections`
- `p242835887_inspection_items`

Required outputs:
- evidence exists
- findings are itemized
- downstream route is explicit

Primary owner:
- `pushingcap_orchestrator`

#### 5. `handoff`

Purpose:
- route the result into repair, parts, transport, DMV, or report-only closeout

Primary records:
- `tickets`
- `tasks`
- `deals` or `quotes` when needed

Required outputs:
- chosen next lane
- responsible execution record
- if work continues, a live downstream ticket or task pack

Primary owner:
- `pushingcap_orchestrator`

#### 6. `closeout`

Purpose:
- deliver the result and preserve the lesson

Primary records:
- `tickets`
- `tasks`
- memory lane

Required outputs:
- client gets findings and next-step recommendation
- operational memory is written back

Primary owner:
- `pushingcap_sick_memory_dude`

Supporting owner:
- `google_run_pc_paygate_bridge` when paid work follows

## What Gets Created Automatically

Once this lane is operationalized, the automation layer should generate:

1. one parent control ticket for the inspection lane
2. one inspection shell from each intake
3. one itemized findings set for every evidence packet
4. one downstream handoff ticket or task pack
5. one closeout packet
6. one memory writeback entry

## What To Stop Doing

Do not:

- treat inspections as only a body-shop task
- force one object to hold intake, evidence, findings, and repair all at once
- leave inspection results as one vague note
- create a downstream lane without itemized findings
- skip memory writeback after a good inspection

## Best Next Build Move

The next clean implementation move is:

1. bind the public intake surface to `p242835887_service_requests`
2. bind that intake to `p242835887_vehicle_inspections`
3. generate `p242835887_inspection_items` from findings
4. spawn a downstream task or ticket pack from the inspection result
5. write the result back into memory so the next inspection is smarter

## Bottom Line

`PushingInspections` is the evidence engine for the automotive control plane.

The platform wins when every inspection can answer:

- what is the asset
- what is wrong
- what is proven
- what must happen next
- which lane now owns the work
